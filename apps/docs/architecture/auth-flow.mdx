title: Auth flow
description: Employer wallet JWT vs contractor email OTP session tokens.

## Employer

`POST /api/auth/employer`: JSON body validated by `employerAuthBodySchema` (`walletAddress`, `message`, `signatureBase58`). `verifyWalletMessageSignature` (`lib/auth.ts`) verifies detached Ed25519 against the claimed pubkey. Success → HS256 JWT (`JWT_SECRET`), claims include `walletAddress`, exp ~7 days.

Protected employer routes expect `Authorization: Bearer <token>` and `verifyEmployerJwt` compares payload wallet to the Blink’s employer row (e.g. create blink, prepare fund, confirm escrow).

## Contractor

Email verification uses short-lived OTP codes stored in Redis (`lib/contractorOtp.ts`), keyed by Blink id. `POST /api/blinks/[blinkId]/contractor/send-code` validates the email matches the Blink’s `contractorEmail` and sends a code via Resend (`lib/email.ts`). `POST /api/blinks/[blinkId]/contractor/verify-code` checks the code, allocates or reuses a `Credential` row and Solana wallet for that email, transitions `PENDING` → `OPENED` when appropriate, and issues a contractor session JWT (`SESSION_SECRET`, `typ: contractor`, `blinkId`, `walletAddress`) via `signContractorSessionToken`.

`POST /api/blinks/[blinkId]/claim` calls `verifyContractorSession` and checks session wallet matches `Blink.walletAddress`.

## Guards

`ensureBlinkContractorAllowed` (`lib/blinkGuards.ts`) rejects contractor actions when the Blink is terminal or past `expiresAt`.
